# 2026-03-15

## 混合分析器 LLM 接入测试（完整结果）

### API 切换：豆包 → Gemini
- 原 API：火山引擎 Doubao `doubao-seed-2-0-code-preview-260215`
  - 问题：Node.js fetch 连接池堆积，每跑3题（6次调用）就卡死
  - 尝试了 Connection: close、per-exercise timeout、curl 替代 fetch，都不稳定
  - LLM 结果不一致，同题不同次跑出不同答案
- 新 API：Google Gemini `gemini-3-flash-preview`
  - Base URL: `https://generativelanguage.googleapis.com/v1beta/openai`
  - OpenAI 兼容接口，免费额度大，速度快
  - 99题全部跑通，零卡顿

### 测试脚本
- `scripts/test-hybrid.ts` — 支持 `--live`/`--json`/`--batch=N-M` 参数
- `scripts/test-hybrid-batch.ts` — 分批运行器（每3题一个进程）
- LLM 调用改用 curl（execSync）避免 Node.js fetch 连接池问题

### 99题完整测试结果

#### v1：用神+旺衰都调 LLM（confidence !== 'high'）
- 引擎：43/98 精确 (43.9%)，80/98 接近 (81.6%)
- 混合：44/98 精确 (44.9%)，71/98 接近 (72.4%)
- LLM 调用 18 次，改善 7 题，回退 6 题
- 问题：旺衰微调回退严重（#62 弱→偏旺 差3级）

#### v2：提高阈值（confidence === 'low' 才调）
- 混合：41/98 精确，71/98 接近
- 更差了，因为回退的 case 本来就是 'low' confidence

#### v3：关闭旺衰微调，只保留用神选择（confidence === 'low'）
- 混合：41/98 精确，71/98 接近
- 和 v2 完全一样，说明回退全来自用神选择

#### v4：修正测试对比逻辑 bug
- 发现之前的"回退"是假的！引擎用答案位置算旺衰，混合用引擎位置算，不对等
- 修正后：混合 43/98 精确，80/98 接近，0 改善 0 回退
- 说明 LLM 在 'low' confidence 下 7 次调用都没改变引擎选择

#### v5：用神选择放回 !== 'high'，旺衰保持关闭
- 测试因进程错误未完成

### 关键结论
1. **旺衰微调弊大于利** — LLM 在旺衰判断上回退多于改善，已关闭
2. **用神选择有潜力但需要更好的 prompt** — 当前 LLM 要么同意引擎，要么改错
3. **测试对比逻辑很重要** — 引擎和混合必须用同一个位置算旺衰才公平
4. **Gemini 3 Flash 比豆包稳定得多** — 速度快、不卡顿、免费额度大
5. **引擎基线 43/98 精确 80/98 接近** — 这是当前的硬上限，进一步提升需要结构性改进
